ADR-001: Architecture Review

Status​

Accepted — Implemented as part of the Prototype Public Release milestone.

Context​

After completing the initial prototype, an architecture review identified several areas requiring improvement before public release:

  • Error handling was scattered across anyhow usage without typed domain errors
  • Entity deduplication relied on name-only keys, causing false negatives and merges
  • No retry logic for transient LLM and embedding failures
  • Ingestion pipeline lacked proper update semantics (overwrites vs. merges)
  • Relations were treated as immutable, missing lifecycle and confidence updates
  • Entities lacked type information for filtering and search refinement

These issues affected correctness, reliability, and usability at scale.

Decisions​

R1: Typed Error Hierarchy​

Decision: Replace anyhow with thiserror for domain-specific error types.

Rationale: Type-safe error handling at boundaries, better error context, and cleaner error propagation in async code.

Implementation:

  • Create ContextKeeperError enum in core with variants:
    • Extraction — Entity or relation extraction failed
    • Embedding — Vector embedding generation failed
    • Storage — SurrealDB operation failed
    • Ingestion — Ingestion pipeline error (orchestration)
    • Config — Missing or invalid configuration
    • Validation — Input validation failed
  • Each variant includes context (source, operation, details)
  • Implement From<T> for interop with upstream crates
  • Use in public APIs; internal code may use anyhow for prototyping

R2: Retry & Validation​

Decision: Add exponential backoff retry logic for transient LLM and embedding calls. Validate all inputs at API boundaries.

Rationale: Network calls are unreliable; retries improve robustness. Early validation prevents garbage-in, garbage-out.

Implementation:

  • Add retry module with configurable exponential backoff (max 3 retries, 100ms–1s delays)
  • Retry on transient errors (rate limits, timeouts, 5xx)
  • Don't retry on permanent errors (400, 401, invalid input)
  • Validate in add_memory(), search(), and other public entry points:
    • Non-empty episode text
    • Valid namespace (alphanumeric, 1-64 chars)
    • Entity/relation counts within bounds
  • Return typed ValidationError on failure

R3: Composite Entity Identity​

Decision: Move from name-only upsert key to composite key: (name, type, namespace).

Rationale: Multiple entities can share the same name (e.g., "Apple" is a company and a fruit). Composite keys prevent false merges and enable type-specific filtering.

Implementation:

  • Entity record gains entity_type field (person, organization, location, event, product, service, concept, file, other)
  • Upsert by (name, entity_type, namespace) instead of name alone
  • Schema migration: backfill existing entities with type other, ensure uniqueness constraint
  • Search can filter by entity type for precision
  • Entity extractor (LLM) now assigns types during extraction

R4: Memory Update Pipeline​

Decision: Implement proper semantics for entity updates during ingestion (merge summaries rather than overwrite; track changes via IngestionDiff).

Rationale: Entities accumulate evidence over time. Overwriting loses information. Tracking changes helps audit and incremental updates.

Implementation:

  • When upserting an entity, merge summaries:
    • Extract key facts from old and new descriptions
    • Combine, deduplicate, and rank by confidence
    • Produce single coherent summary
  • Return IngestionDiff with:
    • Created entities (new records)
    • Updated entities (summary merged, confidence adjusted)
    • Unchanged entities
    • Confidence deltas per entity
  • Allow consumers to log changes, trigger notifications, or update external systems

R5: Relation Lifecycle​

Decision: Treat relations as first-class objects with lifecycle (create, merge, prune). Apply same temporal tracking as entities.

Rationale: Relations evolve: confidence increases with corroborating evidence, decreases if contradicted, and fade if not reinforced.

Implementation:

  • Relation record gains valid_from, valid_until, confidence (0–1)
  • Create new relations with initial confidence (e.g., 0.7)
  • Merge duplicates by increasing confidence (additive: 0.7 + 0.5 → 0.85)
  • Prune relations with confidence below threshold (0.2) after TTL (e.g., 90 days)
  • Timestamp updates on edits; include in temporal queries
  • SurrealDB changefeeds track relation lifecycle for audit

R6: Entity Type Enrichment​

Decision: Support typed entities (person, organization, location, event, product, service, concept, file, other). Enable filtering by type in search.

Rationale: Entity types enable finer-grained search (find people vs. companies), better ranking (bias toward relevant type), and domain-specific reasoning.

Implementation:

  • Define enum: EntityType { Person, Organization, Location, Event, Product, Service, Concept, File, Other }
  • Entity record stores entity_type (see R3)
  • Entity extractor (LLM) outputs type; default to Other if uncertain
  • Search API accepts optional entity_types filter
  • Results grouped by type; boost ranking for type-relevant matches
  • CLI/MCP tools display type in output

Consequences​

Benefits​

  • Better Error Handling — Typed errors catch problems early; callers can react intelligently
  • Reliability — Retry logic reduces transient failures; validation prevents invalid state
  • Correct Entity Identity — Composite keys prevent false merges; entities can be disambiguated
  • Audit Trail — IngestionDiff and soft deletes enable full history
  • Type Safety — Entity and relation types enable filtering, ranking, and domain-specific logic
  • Temporal Correctness — Relations tracked with valid_from/valid_until; point-in-time queries work

Trade-Offs​

  • Slight performance overhead for LLM retries (compensated by reduced errors)
  • Schema migration required for existing deployments
  • Entity extractor must produce types (adds LLM latency)
  • Relation merging logic adds complexity to ingestion

Implementation Status​

These decisions are implemented across the following Linear issues:

  • FZ-59 — Typed error hierarchy (R1)
  • FZ-57 — Retry logic and validation (R2)
  • FZ-58 — Composite entity identity (R3)
  • FZ-12 — Memory update pipeline (R4)
  • FZ-13 — Relation lifecycle (R5)
  • FZ-14 — Entity type enrichment (R6)

See the project milestone plan for detailed task breakdowns and dependencies.
